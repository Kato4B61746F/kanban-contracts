# ADR-001: ポリレポ戦略

- 日付: 2026-05-08
- ステータス: 採用

## 文脈

kanban プロジェクトはバックエンド・Web・MCP・iOS・Android・インフラの 6 種類のコードベースから成る。
これらをモノレポにまとめるか、個別リポ (ポリレポ) にするかを決める必要があった。

## 決定

**ポリレポを採用する**。各コードベースは GitHub の個人アカウント `Kato4B61746F` 直下に独立リポとして置く:

- `kanban-contracts` — 契約 (本リポ)
- `kanban-api` — Hono バックエンド
- `kanban-infra` — Terraform
- `kanban-web` — Next.js
- `kanban-mcp` — MCP サーバー
- `kanban-ios` — SwiftUI
- `kanban-android` — Compose

## 採用理由

1. **iOS / Android はそれぞれ Xcode / Android Studio との統合が前提**で、モノレポに同居する利点が薄い。
2. **リリースサイクルが独立**することが多い (バックエンドだけ毎日デプロイ、モバイルは審査ありで月次など)。
3. **学習目的として、SDK 配布・契約駆動・複数リポのバージョン整合といった実務スキルを獲得したい**。
   モノレポではこの経験が得られない。
4. **CI が軽くなる** (リポ単位で並列実行、無関係な変更で全体が走らない)。
5. 将来 Org に移したり、特定のリポだけ別人に渡したい場合の柔軟性。

## トレードオフ

- **横断的なリファクタが面倒**: 複数 PR を順序立てて出す必要がある。これは ADR-002 の contracts リポを「契約のソース」にすることで緩和する。
- **環境構築が複数回**: `Justfile` と `docker-compose.dev.yml` を親ディレクトリに置いて緩和する。
- **依存バージョンの整合**: `kanban-contracts` が SemVer に厳格に従い、各クライアントが追従する仕組みで担保する。

## 不採用案

- **モノレポ (Turborepo + pnpm workspace)**: モバイルが入る時点で利点が大きく目減り。学習目的にも合わない。
- **複数 Org に分散**: 個人開発の規模で過剰。
